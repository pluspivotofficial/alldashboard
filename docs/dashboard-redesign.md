# 介護職応援ポイント ダッシュボード 軽量化・再設計 設計書

- 作成日: 2026-06-19
- 対象: GAS（Google Apps Script）Web アプリで構築されたダッシュボード
- 目的: 肥大化して重くなったダッシュボードを、データフローと指標を整理したうえで「軽いサイト」に作り直す

> この設計書は「まず設計だけ」を確定させるためのものです。実装は本書の合意後に着手します。
> 末尾の「未確定事項（要記入）」が埋まると、実装に必要な情報がそろいます。

## 1. 背景・目的

- これまで ALL ダッシュボードを Claude + GAS で構築してきた。
- 「みたい指標」が整ってきて、表示項目が増えた。
- デプロイ回数が 180 に達した頃から、サイトが重い。
- 改めて **データフロー** と **みたい数値** を精査し、**軽いサイト** に作り直したい。

### ゴール（成功の定義）
1. 閲覧時の表示が体感で速い（目標: 初回表示 2 秒以内）。
2. データ日数が増えても表示速度が落ちない（増分処理）。
3. 指標が整理され、不要なものが落ち、必要なものが見やすい。
4. 今後の修正（Claude での編集）がしやすい構成。

## 1.5 確定したデータフロー（突合パイプライン）

ダッシュボードが表示する数値は、以下の 5 ステップで作られる。

```
① obo-data-tool（GitHub Pages）
   7 媒体（エン派遣 / 自社 / バイトル / マイナビ / e介護転職 / キューメイト / 求人ボックス）の
   CSV を統合 → "Web 経由の総応募" を取得（CSV 出力）
   URL: https://pluspivotofficial.github.io/obo-data-tool/
                 │
                 ▼
② 過去応募データ（Drive）と突合して重複判定 → 新規応募数を割り出す
   （重複 = 再応募、非重複 = 新規）
                 │
③ MCG（CRM）から前日までの全 CSV をダウンロード（Drive 可）
   ※ 電話応募は MCG に直接入力され、Web 総応募（①）には含まれない
                 │
④ 当月分の新規応募について「人選判定」を実施（判定ロジックは別途共有）
                 │
⑤ 人選を含む最新データを Drive にアップロード
                 │
                 ▼
   【突合】 総応募データ ＋ MCG データ ＋ 人選入り MCG データ
                 │
                 ▼
   オフィス単位（都道府県ではなくオフィス）で集計し、3 テーブルを表示
```

### データソース整理

| 記号 | ソース | 置き場所 | 主な役割 |
|------|--------|----------|----------|
| ① | obo-data-tool 統合 CSV | GitHub Pages 出力 → Drive | Web 経由の総応募 |
| 過去 | 過去応募データ | Drive | 重複判定（新規/再応募の切り分け） |
| MCG | MCG 全 CSV（前日まで） | Drive | 電話応募を含む全応募・接触・歩留 |
| 人選 | 人選入り MCG データ | Drive | A/B/C/その他/不明 の人選判定 |

> 重要: **新規応募 = Web 新規（①を過去と突合）＋ 電話新規（MCG）**。電話応募は①に含まれないため MCG 側から補完する。

### 表示する 3 テーブル（オフィス単位 = 行）

**テーブル1: 応募サマリー**

| 列 | 定義（確定） |
|----|--------------|
| 新規応募数 | 当月の新規応募（Web 新規 ＋ 電話新規）。Web は①を過去と突合した非重複分、電話は MCG 由来 |
| 再応募数 | 過去と重複した応募。**同一人物が複数回応募してもユニーク 1 件**として数える（重複排除） |
| 目標の新規応募数 | オフィス別マスタ（スプレッドシート 1 枚）で管理 |
| 着地見込み | **ペース換算** = 当月新規 ÷ 当月経過日数 × 当月総日数 |
| 接触数 | MCG の「接触ステータス」が **接触（電話）/ 接触（フォーム）/ 接触（メール）** のいずれかのレコード数 |

**テーブル2: 人選判定（人数）**

| 列 | 定義 |
|----|------|
| A人選 / B人選 / C人選 / その他 / 不明 | 人選入り MCG データの判定区分ごとの人数 |

**テーブル3: 歩留（応募区分 × ステータス）**

| 応募区分 | ステータス列（MCG） |
|----------|----------------------|
| 当月内応募・新規 | 設定日（新規）/ 実施日（新規）/ 決定日（新規）/ 開始日（新規）が入力済みのレコード数 |
| 2ヶ月以内応募・新規 | 同上の列。応募日が直近 2 ヶ月以内のコホート |
| 再応募 | 設定日 / 実施日 / 決定日 / 開始日（再応募）の列（※再応募用列名は要確認） |

> 「設定 / 実施 / 決定 / 開始」= 各日付列が **非空（入力済み）** のレコードを数える。

## 2. 現状アーキテクチャ（診断）

### 2.1 現在の構成

```
[毎日] 担当者がファイルを Google Drive のフォルダへアップロード
                 │
                 ▼
        GAS Web アプリ（doGet）
                 │  閲覧者がアクセスするたびに…
                 ▼
   フォルダ内の "全ファイル" をループで開く → 解析 → 集計 → HTML 生成
                 │
                 ▼
            ブラウザに表示（重い）
```

### 2.2 重さの原因分析

| # | 原因 | 影響 |
|---|------|------|
| 1 | **ページを開くたびにフォルダ内の全ファイルを読み直して集計している** | 日数（ファイル数）に比例して遅くなる。180 日分で毎回 180 ファイルを開く。GAS の実行時間制限・クォータにも当たりやすい。**これが体感の重さの主因。** |
| 2 | **事前集計（キャッシュ）が無い** | 同じ集計を閲覧のたびに毎回ゼロからやり直している。 |
| 3 | **データ処理と表示が同じ場所（doGet 内）に同居** | 重い処理を「表示の瞬間」に走らせている。分離されていない。 |
| 4 | 表示する指標が増え、計算もクライアント/表示時に寄っている可能性 | 表示時の計算量が増える。 |

> ※「デプロイ 180 回そのもの」は配信を重くしません。重いのはあくまで **毎回フォルダ全走査** です。
> したがって、デプロイ数を減らすことより **処理の作り方を変えること** が本質的な対策です。

## 3. 新アーキテクチャ設計

### 3.1 全体方針

**「集計（データ処理）」と「表示（配信）」を分離する。**

- 集計は **1 日 1 回だけ** まとめて実行（毎日のアップロード後）。
- 表示は **集計済みの軽いデータを 1 回読むだけ**。

```
【改善後】

① 毎日アップロード後、時間トリガーで 1 回だけ集計
      Drive フォルダの "新規ファイルだけ" を解析（増分）
                 │
                 ▼
      1 つの「集計済みデータ（サマリー）」に圧縮して保存
        ・保存先候補: 1 枚のサマリーシート / 1 個の summary.json / CacheService
                 │
② 閲覧者がアクセス
                 ▼
      集計済みサマリーを "1 回だけ" 読む → 即表示（軽い）
```

### 3.2 コンポーネント設計

#### (A) 取込・集計層（GAS：時間トリガー）
- **トリガー**: 毎日 1 回（CSV出力の後、毎朝 9 時台。統合ツールが〜8:00にCSVを出すため。`CONFIG.DAILY_TRIGGER_HOUR`）。
- **増分処理**: 「前回処理済みの日付/ファイルID」を `PropertiesService` に記録し、**新規ファイルだけ** 解析する。日数が増えても処理時間が一定。
- **出力**: 全期間の集計結果を **1 つのサマリー** にまとめて保存。
- 失敗時はメール通知（任意）。

#### (B) キャッシュ／データ層
- 集計済みサマリーを **1 個の JSON** として保存（`CacheService`（最大 6 時間）＋恒久バックアップとして Drive 上の `summary.json` か 1 枚のシート）。
- 閲覧時はこのサマリーを返すだけ。フォルダ全走査は二度としない。

#### (C) 配信・表示層（フロントエンド）
- 集計済みサマリーを読み込み、グラフ/テーブルを描画するだけの軽い画面。
- 重い計算は (A) で済んでいるので、表示は描画のみ。

### 3.3 構成 2 案（方向性の比較）

> **決定: 案1（静的フロント + GAS は JSON API）を採用。**（セクション 10 参照）

| 観点 | 案1: 静的フロント + GAS は JSON API | 案2: GAS 内で完結（事前集計のみ追加） |
|------|-----------------------------------|--------------------------------------|
| 表示HTMLの置き場所 | この GitHub リポジトリ（GitHub Pages） | GAS（HtmlService） |
| GAS の役割 | 集計済み JSON を返すだけ（薄い API） | 集計 + 表示の両方（ただし集計は事前化） |
| 速度 | ◎ 最速（静的配信 + 軽量 JSON 1 回） | ○ 速い（事前集計の効果） |
| 今後の修正しやすさ | ◎ Claude での編集が GitHub 上で完結 | △ GAS エディタ中心 |
| 移行コスト | △ フロント移設が必要 | ◎ 小さい（既存に追記） |
| 推奨度 | **推奨（軽さ・保守性重視なら）** | 移行コストを抑えたいなら |

> いずれの案でも **(A) 事前集計 + (B) キャッシュ** が最大の効き目。まずここを入れるだけで体感は大きく改善します。

## 5. データ契約（集計済みサマリーのスキーマ案）

表示層が読む「集計済みサマリー」の形を決めておくと、集計層と表示層を独立に作れます。
3 テーブル仕様に合わせた **確定版スキーマ**（`dashboard/index.html` のモックもこの形を消費）:

```json
{
  "generatedAt": "2026-06-19T06:00:00+09:00",
  "month": "2026-06",
  "daily": [
    { "date": "2026-06-18", "new": 11, "re": 2, "total": 13 },
    { "date": "2026-06-19", "new": 9,  "re": 3, "total": 12 }
  ],
  "offices": [
    {
      "office": "新宿オフィス",
      "prefectures": ["東京都"],
      "overview": {
        "newApplications": 42,
        "phoneApplications": 6,
        "reApplications": 8,
        "targetNew": 60,
        "forecast": 55,
        "contacts": 30,
        "newAB": 14,
        "reAB": 3
      },
      "selection": { "A": 5, "B": 10, "C": 8, "other": 3, "unknown": 2 },
      "funnel": {
        "currentMonthNew":   { "set": 20, "done": 15, "decided": 8, "started": 5, "ab": 9 },
        "within2MonthsNew":  { "set": 10, "done": 7,  "decided": 4, "started": 2, "ab": 5 },
        "reApplication":     { "set": 6,  "done": 4,  "decided": 2, "started": 1, "ab": 2 }
      }
    }
  ]
}
```

ポイント:
- `daily`: **全体（全オフィス合算）の日次推移**。一覧ビューのグラフ用（積み上げ棒＝新規＋再応募、線＝総応募）。
- `overview.newAB` / `overview.reAB`: 新規・再応募に含まれる **A+B 人選の人数（参考値）**。
- `funnel.*.ab`: 各応募区分の **A+B 人選の人数（参考値）**。
- オフィス 1 件 = 1 オブジェクト。合計行は表示側で算出。
- すべて **集計済みの小さな数値** のみ（生ファイル・個人データは渡さない）。
- 表示はこの JSON をテーブル／グラフに流すだけ。差し替え＝ `fetch()` 先を GAS の API に向けるだけ。

## 6. 移行手順（段階的・低リスク）

各ステップは独立して効果が出るので、上から順に入れれば途中でも体感が改善します。

1. **【最優先】事前集計の導入（案1/案2 共通）**
   - 時間トリガーで全期間を 1 回集計 → サマリー保存。
   - 閲覧時はサマリーを読むだけに変更。→ **ここで重さの大半が解消。**
2. **増分処理化**
   - 処理済みファイルを記録し、新規分だけ解析。→ 将来も速度維持。
3. **キャッシュ追加**
   - `CacheService` でサマリーを保持。→ 連続アクセスが即時。
4. **指標の棚卸し反映**（セクション 4）
   - 不要指標を落とし、必要指標をサマリーに追加。
5. **（案1 を採る場合）フロント静的化**
   - 表示 HTML をこのリポジトリ（GitHub Pages）へ移設し、GAS は JSON API 化。

## 8. 集計ロジック仕様（確定）

GAS 集計層がサマリー JSON を生成するときの計算ルール。

### 8.1 マスタ（スプレッドシート）

| マスタ | 役割 | 実際の列 |
|--------|------|----------|
| 都道府県↔オフィス 対応表 | 突合キー。**全データを都道府県でオフィスへ振り分ける** | `都道府県`, `オフィス` |
| オフィス別 目標マスタ | テーブル1 の「目標(新規)」供給元 | `オフィス`, `目標` |

> 対応表は一部の都道府県のみ収録（北海道・中国四国などは未収録）。未収録の都道府県の応募は現状どのオフィスにも振り分けられない（要確認）。

### 8.1.1 新規/再応募の判定（確定・累積シートで重複判定）

総応募シートは **5/1 以降＋過去応募をすべて累積** する運用（obo-data-tool が前日分を自然に追記）。
区分は列に頼らず、**集計側で電話番号の重複判定**を行う。

- シートを応募日昇順に走査し、**電話番号の初出＝新規（1）／2回目以降＝再応募（0）**。
- `重複応募`・`有効応募`列は使用しない。
- 再応募は **電話番号でユニーク化**（例: 同一人物が3行 → 再応募1）。
- 累積データがあるため、**2ヶ月以内コホート**も初回応募日で正しく判定できる。

### 8.1.2 電話応募（確定）

- 電話応募は総応募CSVに無く、MCGにのみ存在する。
- **定義: MCG（⑤）にあって総応募に無い電話番号 = 電話応募**（当月・電話番号でユニーク）。
- 電話応募は **新規応募に加算**し、歩留でも新規扱い。
- **電話応募数**は内訳指標として別途表示（`overview.phoneApplications`）。

### 8.2 オフィスへの振り分け（突合キー）

- **基準キーは「都道府県」**。
- **MCG データ**: 都道府県のみ記載 → 対応表で `都道府県 → オフィス` に変換。
- **総応募（①）**: オフィス名が記載 → そのまま使用（対応表で都道府県と整合チェック）。
- 1 オフィスが複数都道府県を持つ場合あり（例: 大阪オフィス = 大阪府・兵庫県）。対応表は `都道府県 → オフィス`（多対一）。

### 8.3 指標の算出ルール

| 指標 | 計算 |
|------|------|
| 新規応募数 | 総応募で電話番号が初出（当月）＋ 電話応募（MCGにあり総応募に無い、当月・電話ユニーク）。 |
| 電話応募数 | 上記の電話応募分（内訳）。`overview.phoneApplications`。 |
| 再応募数 | 当月の応募のうち電話番号が2回目以降のものを **電話番号でユニーク化**（10回でも1）。 |
| 目標(新規) | 目標マスタ（`オフィス`/`目標`）の該当オフィス値。 |
| 着地見込み | `round(当月新規 ÷ 当月経過日数 × 当月総日数)`（ペース換算・暦日）。 |
| 接触数 | MCG の接触ステータスが「接触」始まり（接触 (電話)/(フォーム)/(メール)）のレコード数。当月応募分。 |
| 人選 A/B/C/その他/不明 | 人選入り MCG データの判定区分ごとの人数。 |
| 歩留 設定/実施/決定/開始 | 各ステージの**日付列が当月に入っている**レコード数（非空ではなく「その日付＝当月」でカウント）。参照列は全コホート共通で「設定日（新規）／実施日（新規）／決定日（新規）／開始日（新規）」。新規/再応募の区分は総応募データから判定。 |
| 参考: A+B 人選（newAB/reAB/funnel.ab） | 各集合のうち人選が A または B の人数。 |

### 8.4 歩留コホートの定義

| コホート | 応募者の条件（応募日・区分） |
|----------|------|
| 当月内応募・新規 | 新規 かつ 応募日が当月 |
| 2ヶ月以内応募・新規 | 新規 かつ 応募日が **当月を含む直近 2 ヶ月**（当月内応募を内包） |
| 再応募 | 初回応募日が当月より前（過去に応募済みで再来した集合） |

> 各セルは「対象コホートの応募者」かつ「その（新規）ステージ日付が**当月**」の件数。
> 当月内応募・新規 ⊂ 2ヶ月以内応募・新規（当月応募者は両行に計上される）。

## 9. GAS 集計層の設計（実装方針）

```
[毎日CSV出力後] 時間トリガー（毎朝 9 時台）
  1. 当月のマスタ（対応表・目標）を読み込む
  2. ① 総応募 CSV を読み、過去応募データと突合 → Web 新規 / 再応募を判定
  3. MCG CSV を読み、電話新規・接触・歩留・（再応募の本人ユニーク化）を集計
  4. 人選入り MCG データで A/B/C/その他/不明、A+B 参考値を集計
  5. 都道府県 → オフィスに振り分けてオフィス単位で合算
  6. 日次（全体）の new/re/total を集計
  7. サマリー JSON（セクション 5 のスキーマ）を生成
  8. 保存: CacheService（高速）＋ Drive 上 summary.json（恒久）
  9. （案1 の場合）doGet は summary.json を返すだけの薄い API
```

- **増分処理**: 処理済みファイル ID / 最終処理日を `PropertiesService` に記録し、新規分のみ解析。
- **月またぎ**: 当月 + 直近 2 ヶ月分を保持すれば 2ヶ月コホートも計算可能。
- **個人情報**: サマリーには集計済みの数値のみを出力し、氏名・連絡先など生データは出さない。
